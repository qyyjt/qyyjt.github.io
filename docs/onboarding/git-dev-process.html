# 开发升级流程

一般需求开发时会有如下流程：

  1. 在本地开发，并更新到 dev 环境自测、测试、UI 样式校对问题修改
  2. 发布上线时，提交到 test 环境
  3. 中台人员用 test 环境的代码，发布 uat
  4. uat 无问题，发布外网

所以升级的时候，我们的工作并不多，只需要把打包后的代码搞到 test 环境的服务器上就好了。

# git 开发流程

我们在项目开发中，会用到如下几个不同分支。

分支 介绍
dev 开发环境，升级时合并到 master
master 代码会自动编译到 test 环境
feature/* 功能分支
bugfix/* bug 分支，用于修复外网 bug

开发中，常见的有主要有如下几个场景

  • 内网开发新功能
  • 修复已发布到外网的 bug

假如，我们有一个需求号为 1650，这时候需要创建一个功能分支 feature/#1650。或者禅道的 bug 号是 123，则创建bugfix/#123分支。

dev 分支会在升级时合并到 master 分支，所以我们可以从 dev分支去创建功能分支。

在开发过程中，若你需要在 dev 环境查看效果，可以直接使用 gulp ，把打包后的文件更新到 dev 服务器，或者手动更新。此时无需合并到 dev 分支，因为这个时候，你并不知道这个需求会在何时上线；需要等到升级的时候，以禅道看板 (opens new window)中的升级单为准，若你的需求或 bug 在此次升级范围内，再合并到 dev 分支

这样或许有一个问题，就是如果两个人同时在这个项目中，开发不同的需求，那么 gulp 的时候，就会互相覆盖掉对方的代码，对此可以双方商量一下，看哪方需要测试，就哪一方 gulp 一下

TIP

请避免在 dev 分支直接开发新功能。

  • 第一种情况，还没有 commit

可以先执行git stash，将代码先暂存起来，然后切换到功能分支，执行git stash pop将暂存的代码释放出来。然后在功能分支开发即可。

  • 第二种情况，已经在 dev 分支上 commit 过

首先，把 dev 分支上该需求的提交过滤出来，复制所有关于该需求提交的 hash 值，注意多个哈希值是按时间的正序排列，即最早提交的 hash 在前，后提交的 hash 在后。如果你用的是 Webstorm，它会自动帮你排好哈希顺序。

紧接着，切换到功能分支，执行 cherry-pick，将刚才选中的提交 hash 全部 cherry-pick 到功能分支上。

git checkout feature/xxx
git cherry-pick -x hash1 hash2 ...

如果出现冲突，请解决冲突，解决完执行 git cherry-pick --continue 继续，直到结束。

之后，在功能分支上开发即可。

# 项目升级流程

升级时，把需要升级的功能对应的分支合并到 dev，再使用 merge request (opens new window) 的方式把 dev 合并到 master 。等待编译完成时，即可在 test 环境自测

若代码在 svn 上，或者再 git 上提交后没有效果，可以使用手动提交的方式。
